# Project Overview

This application is an AI-powered paid search campaign builder. Given a landing page URL, the agent analyzes the page content with particular focus on above-the-fold content, generates tightly themed ad groups with keywords and RSA ad copies, exports the campaign structure as an Excel workbook for business user review, and generates a Google Ads Script that the user runs directly inside their Google Ads account to launch the campaign. Bing Ads is a secondary launch target.

## Key Components

- **AI Engine**: Claude via the Anthropic API
- **Landing Page Fetching**: Puppeteer MCP handles page fetching and rendering
- **Campaign Export**: Excel workbook generation via openpyxl (Summary, Keywords, Ads tabs) -- universal baseline format. Google Sheets upload can be added as an optional enhancement later
- **Campaign Launch**: Generate Google Ads Scripts that the user copies and pastes directly into Google Ads to launch the full campaign

## Standard Workflow

- Before writing any code, think through the problem, read the codebase for relevant files, and write a plan to Plan.md
- The plan must include a checklist of todo items that get checked off as each is completed
- Plan.md is a persistent file — never delete or overwrite it between sessions. It is the running record of progress
- At the start of each session, re-read Plan.md to restore context before doing anything else
- Before beginning any work, check in and wait for the plan to be verified
- Once verified, work through todo items one at a time, marking each complete as you go
- At every step, give a brief high-level explanation of what changes were made and why
- Make every task and code change as simple as possible — every change should impact as little code as possible
- Favor short modules, short methods, and short functions — name everything clearly
- Build one step at a time, working incrementally and testing after every chunk of code developed
- Never make large or complex changes in a single step — if a change feels big, break it down further
- Always use the latest stable APIs — check documentation if uncertain
- No emojis anywhere in code, comments, or responses

## Technical Decisions

### Stack

- **Frontend**: Next.js (latest) with TypeScript and Tailwind CSS
- **Backend**: Python FastAPI
- **Database**: PostgreSQL via Supabase (provides auth, storage, and realtime out of the box)
- **ORM**: SQLAlchemy with Alembic for migrations
- **Deployment**: Docker Compose with separate containers for frontend and backend
- **Package Manager**: uv for Python dependencies inside Docker containers

### Frontend Conventions

- Use Next.js App Router for all routing
- Use modals over native JS popups for all user interactions
- State management: Zustand for global client state, TanStack Query for all async server state
- Components live in `/components`, pages in `/app` following App Router conventions
- Never use `var` — use `const` and `let` only
- Prefer `async/await` over promise chains

### Backend Conventions

- API routes follow RESTful conventions
- FastAPI routers organized by domain (campaigns, keywords, ads, etc.)
- All endpoints return consistent JSON response shapes with error handling
- Use type hints throughout all Python code
- Use Pydantic models for all request/response schemas

### API Pattern

- All API calls from frontend go through `/app/api` route handlers (BFF pattern), never directly to FastAPI from client
- Exceptions allowed for simple read-only endpoints where the proxy would be pure boilerplate

### Environment Variables

- All secrets and API keys stored in `.env.local` (never committed) — verify `.env.local` is in `.gitignore` before anything else
- `.env.example` maintained with all required keys and placeholder values for documentation purposes
- Backend secrets passed to FastAPI container via Docker Compose environment block
- Use different API keys for development vs. production
- Required keys: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `GOOGLE_ADS_API_KEY`, `BING_ADS_API_KEY`, `ANTHROPIC_API_KEY`

### MCP Configuration

- **Puppeteer MCP** — handles all landing page fetching and rendering, including JavaScript-heavy pages. Required for above-the-fold content analysis. Used both at runtime and during development.
- Puppeteer MCP must be configured and verified before any agent pipeline work begins
- Google Sheets MCP is not currently used -- campaign export uses server-side Excel generation via openpyxl. Google Sheets integration can be added as an optional upload target later.

## Testing and Debugging

- Try one test at a time. Be methodical
- Don't jump to conclusions
- Don't apply workarounds
- Reproduce the issue consistently before attempting a fix, where applicable
- Identify root cause before fixing. Prove the cause with evidence — don't guess — then fix
- Unit tests: Vitest (preferred over Jest for better ESM support and speed in Next.js ecosystem)
- Backend tests: pytest
- E2E tests: Playwright

## Quality Guidelines

- Test coverage must remain above 90%
- All tests must pass before committing
- ESLint and Prettier must pass without errors
- Before declaring any task complete, always run the linter and full test suite and confirm both pass

## Monitoring, Alerting and Logging

### Logging

- Use Pino for structured JSON logging in the Next.js frontend and API layer
- Use Python's built-in logging module with a JSON formatter for FastAPI
- Both log streams fed into a single unified stream via Docker Compose
- All log entries must include: timestamp, level, service, message, and context
- Never log secrets, API keys, session tokens, or PII

### Error Tracking

- Use Sentry for both Next.js and FastAPI — SDKs available for both
- Sentry catches and reports all unhandled exceptions with full stack traces
- Configure Sentry alerts to notify via email or Slack on new error types

### API Call Logging

- Every external API call must log: request details, response status, and latency
- Applies to all external services: Anthropic, Google Ads, Bing Ads
- Critical for debugging agent chain failures and tracking API costs over time

### Agent Chain Logging

- The agent pipeline makes multiple chained calls — landing page fetch, page analysis, keyword generation, ad copy generation, Excel export, Ads Script generation
- Each step in the chain must log its start, completion, and any failure with enough context to identify exactly where in the chain a failure occurred
- Log the full chain execution time end to end

### Alerting

- Keep alerting simple in early development — Sentry email or Slack notifications on new error types is sufficient
- Do not over-engineer alerting until there are real users in production

## Security Requirements

### API Key Management

- Set spend caps and rate limits on all AI API keys (Anthropic) directly in their dashboards — turns mistakes into minor inconveniences rather than expensive incidents
- Scope all API keys to minimum required permissions — a Google Ads key should be able to create campaigns, not access billing
- Rotate keys immediately if any are accidentally exposed

### Authentication and Authorization

- Use Supabase Auth for all user authentication — do not roll your own
- Enable row-level security (RLS) in Supabase from the start — each business user should only ever see their own campaigns, keywords, and ad copies
- All Next.js API route handlers must validate the user session before processing any request
- Never trust client-supplied user IDs — always derive identity from the verified session server-side
- All API endpoints require authentication except health check endpoints — enforce this as a default when scaffolding any new FastAPI route

### Input Validation

- Validate and sanitize all landing page URLs before the agent fetches them — first external-facing attack surface
- Validate all inputs on both frontend and backend — never rely on frontend validation alone
- Reject malformed or unexpected API payloads with a 400 response before any processing occurs

### Data Safety

- All external communication over HTTPS only — no exceptions
- Supabase connection must use the service role key only in server-side contexts, never exposed to the client
- Use SQLAlchemy parameterized queries — never construct raw SQL strings
- Never return internal database models directly in API responses — always map to explicit response schemas to avoid exposing unintended fields

### CORS and Rate Limiting

- Configure explicit CORS settings on FastAPI — only allow requests from the frontend origin
- Rate limit API endpoints to prevent abuse, especially the agent pipeline endpoint which triggers expensive AI calls

### Docker Security

- Run containers as non-root users

### Security Logging

- Log all security events — authentication failures, authorization violations, and unexpected access patterns — as distinct log entries for audit purposes
- Use service accounts for all automation credentials, never personal OAuth tokens — personal tokens expire and will silently break the agent pipeline
